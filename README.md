# B2B Synthetic Document Dataset Pipeline

A zero-cost, industrial-grade Python pipeline that generates **40,001 documents** — 20,000 Purchase Orders + 20,000 Tax Invoices + 1 SQLite Master Ground-Truth Log — for training spatial OCR and multimodal Document Understanding models.
python3 main.py --industry "Automotive Fasteners" --count 50 --workers 2

---

## Module Documentation

### `pipeline/bootstrap.py` — Pass 1

Calls the Ollama `/api/generate` endpoint with `temperature=0.0` and a **strict JSON Schema** to produce 1,000 industry-specific line items.

- **10 batches × 100 items** per batch
- Each item: `description`, `hsn_code` (8-digit), `unit`, `unit_cost_inr`, `gst_rate` (5/12/18/28), `category`
- Validates HSN format, GST rate enum, and unit of measure
- Falls back to a deterministic pure-Python catalog if Ollama is unavailable
- Saves to `industry_catalog.json`

### `pipeline/data_models.py` — Field Generators

All randomisation is **seeded per document index** (`rng = random.Random(idx * 31337 + 7)`) ensuring full reproducibility.

**Generated fields:**
| Field | Generator |
|-------|-----------|
| Company names | Faker `en_IN` locale |
| Addresses | Faker Indian addresses |
| GSTIN | 15-char plausible format: `{state_code}{PAN}{entity}Z{checksum}` |
| PO Number | `{prefix}-{year}-{seq:05d}` |
| Invoice Number | `{prefix}-{year}-{seq:05d}` |
| Dates | Random date 2022–2024; Invoice = PO + 1–14 days |
| Line items | 1–5 items sampled from catalog; qty 1–200 |
| Payment terms | Pool of 10 real B2B terms |
| Terminology | Shuffled from pools (6 buyer, 6 seller, 5 ship-to labels) |

**Financial calculations** (pure Python, no floating-point drift):
- Line total = `qty × unit_cost` (rounded to 2dp)
- CGST = SGST = `taxable × gst_rate/2 / 100` (intra-state)
- IGST = `taxable × gst_rate / 100` (inter-state)
- Grand total = `subtotal + total_tax`

### `pipeline/layout_engine.py` — Block-Based Floating Layout

**8 layout variants** × **8 colour themes** × **4 handwritten fonts** = 256 distinct visual configurations.

Block rendering order is shuffled per variant:
```
Variant 0: seller → buyer → shipping → meta → items → summary → footer
Variant 1: meta → seller → buyer → items → shipping → summary → footer
... (8 total permutations)
```

Floating blocks use CSS `flex: 1 1 220px; flex-wrap: wrap` — they reflow automatically based on content width, simulating different enterprise ERP print layouts.

**Handwritten font injection:** Caveat, Dancing Script, Kalam, or Patrick Hand applied via Google Fonts to:
- PO/Invoice reference numbers
- Dates (all date fields)
- Signatory names
- Sign-off dates

**Red ink stamp overlay:** SVG-based, semi-transparent (12–28% opacity), randomly rotated (-35° to -5°), positioned at random 30–55% canvas overlap. Present in ~55% of POs and ~65% of invoices.

### `pipeline/renderer.py` — Playwright HTML → PNG

- Viewport: 900×1200px
- `deviceScaleFactor: 2.0` → effectively 1800×2400px render
- Waits for `networkidle` to ensure Google Fonts load
- Clips to `.document` element bounding box
- Persistent browser context per worker (no per-doc browser launch overhead)
- Pillow fallback renderer when Playwright unavailable

### `pipeline/degradation.py` — OpenCV Pipeline

**8 degradation profiles** applied deterministically by `doc_index % 8`:

| Profile | Noise σ | Fade | Max Rotation | JPEG Quality |
|---------|---------|------|--------------|--------------|
| clean_laser | 0.010 | 2% | 0.3° | 95 |
| worn_laser | 0.040 | 8% | 0.8° | 85 |
| inkjet_fresh | 0.020 | 3% | 0.5° | 90 |
| inkjet_old | 0.060 | 15% | 1.2° | 75 |
| fax_quality | 0.100 | 20% | 1.8° | 60 |
| archive_scan | 0.080 | 12% | 1.0° | 70 |
| high_res_clean | 0.005 | 1% | 0.1° | 98 |
| low_ink | 0.030 | 25% | 0.6° | 80 |

Pipeline stages per image:
1. Gaussian noise (CCD sensor grain)
2. Pixel fading (dark-pixel stochastic lightening)
3. Brightness/contrast adjustment (±12% variance)
4. Small rotation (scanner misalignment)
5. JPEG re-compression (lossy artefacts for lower-quality profiles)

### `pipeline/database.py` — SQLite Master Log

Two tables with WAL journal mode for concurrent writes:

**`documents`** — one row per image
```sql
id, doc_type, doc_index, filename, po_number, invoice_number,
json_payload (full JSON), buyer_name, seller_name, buyer_gstin,
seller_gstin, is_interstate, subtotal, total_tax, grand_total,
num_line_items, layout_variant, degradation_profile, created_at
```

**`document_pairs`** — cross-reference PO ↔ Invoice
```sql
pair_id, doc_index, po_doc_id, inv_doc_id, po_number,
invoice_number, grand_total, terminology_po_buyer,
terminology_inv_seller, created_at
```

### `pipeline/assembler.py` — Multiprocessing Coordinator

Uses `concurrent.futures.ProcessPoolExecutor`:
- Each worker subprocess handles a **shard** of document indices
- Each worker runs its own `asyncio` event loop for Playwright
- Workers write PNG files and log to SQLite concurrently (WAL prevents contention)
- Progress logged every 100 documents per worker
- Partial failures logged without stopping the pipeline

## Output Format

### PNG Images
- Location: `output/purchase_orders/po_NNNNNN.png`, `output/tax_invoices/inv_NNNNNN.png`
- Resolution: ~1600×2200px (varies by content height)
- Format: PNG (lossless after degradation)
- Naming: zero-padded 6-digit index

### SQLite Database
```bash
# Query example: all invoice pairs above ₹1 lakh
sqlite3 output/master_ground_truth.db \
  "SELECT po_number, invoice_number, grand_total
   FROM document_pairs
   WHERE grand_total > 100000
   ORDER BY grand_total DESC
   LIMIT 10;"

# Retrieve full JSON payload for document verification
sqlite3 output/master_ground_truth.db \
  "SELECT json_payload FROM documents WHERE filename='po_000042.png';"
```

## Extending the Pipeline

**Add a new document type:** Create a new dataclass in `data_models.py`, a new renderer in `layout_engine.py`, and update `assembler.py`'s `process_one()`.

**Change catalog size:** Modify `BATCH_SIZE` and `TARGET_ITEMS` in `bootstrap.py`.

**Add more degradation effects:** Extend `pipeline/degradation.py` — the pipeline is modular; add new steps after the existing ones.

**Custom industries:** Just change `--industry` — the LLM generates a fresh domain-specific catalog automatically.
